- Added the **directive-close** PreToolUse matcher (#490): a GitHub close keyword (`close`/`closes`/`closed`, `fix`/`fixes`/`fixed`, `resolve`/`resolves`/`resolved`, case-insensitive) followed by `#N` is now blocked when `#N` is a **Directive** Issue — in either the inline `--body`/`-b` of `gh pr create`/`gh pr edit` or a commit message (subject **or** body). At merge GitHub would auto-close the referenced Issue, which for a Directive bypasses `/complete-directive`'s signal-evidence gate (SPEC §5.13) — the live regression where PR #489's `closes #477 signal 3` force-closed Directive #477 while its signal 4 was still pending. Execution Issues are unaffected (`Closes #<execution>` is the normal resolving trailer) and a non-close mention (`Refs #N`, `advances #N`) always passes; the matcher fails open per `#N` on an unresolved type. The inline body is read; `--body-file`/`-F` is a documented residual (#490)

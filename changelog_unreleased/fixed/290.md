- Hardened the `merge-strategy` matcher's command parsing into a gh-flag-aware token walk (`parse_gh_merge_argv`, shell-aware via `shlex` with a `read -ra` fallback), closing three matcher-vs-gh mismatches from #288's security review (all below SPEC §6.1's guardrail threshold, but real): (A) a `--merge` appearing inside a `--body`/`--subject` *value* is no longer read as the strategy — strategy is the explicit flag token, not a substring, so `gh pr merge N --squash --body "see --merge"` correctly blocks; (B) the PR is the first *positional* token (or a `.../pull/N` URL), skipping value-taking flags' values, so `gh pr merge --body 7 --squash 5` resolves PR 5 (not 7); (C) the short merge flag `-m` is now recognized as compliant (it was wrongly blocked), and `-s`/`-r` short squash/rebase are blocked. The core §5.7.1 enforcement (block squash/rebase/bare to the default branch, allow `--merge`, allow squash on a non-default base) is unchanged; this only sharpens how strategy and PR are read from the command.

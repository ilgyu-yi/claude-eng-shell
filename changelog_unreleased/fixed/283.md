- Hardened the `resolve_gh_issue_target` selector parser (deferred follow-up from #276): it now tokenizes **shell-aware** via python3 `shlex` (mirroring `check_destructive_args`, with a `read -ra` fallback), so a quoted flag value containing an interior digit — `gh issue edit --body "fixes 99 things" 7 …` — no longer has its `99` mistaken for the positional issue selector (it correctly resolves `7`); and it normalizes a host-prefixed repo (`--repo github.com/o/r` → `o/r`) so the downstream `owner/name` derivation isn't `github.com`/`r` (which had caused a wrong-repo / fail-open lookup). On an unparseable command (unclosed quote — which would not execute in a real shell anyway) the helper emits an empty issue so the caller fails open. Both limitations were fail-safe before this fix (wrong-same-repo / fail-open, never a cross-target bypass); this closes them for accuracy. (#283)
